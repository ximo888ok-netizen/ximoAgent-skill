# project-map-skill

> 让 AI 编程智能体先看地图再动手 —— 项目知识图谱生成与检索技能

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](https://opensource.org/licenses/MIT)
[![Python 3.8+](https://img.shields.io/badge/python-3.8+-blue.svg)](https://www.python.org/downloads/)

## 这是什么

一个 **AI 编程技能（Skill）**，为任意项目自动生成结构化知识图谱（`PROJECT_MAP.md`），并提供轻量检索工具让 AI 按需查询，避免每次都读取全文浪费 token。

**核心理念**：AI 动手改代码前，先查地图了解项目结构，就像程序员先看架构文档再动手一样。

## 解决什么问题

| 痛点 | 没有 project-map | 有 project-map |
|------|-----------------|----------------|
| AI 不了解项目结构 | 盲目猜测，经常改错地方 | 先 `list` 再精准查章节 |
| 读取项目文档费 token | 每次读全文 ~6000 token | 按需检索 ~300 token，省 80-95% |
| 上下文压缩后丢信息 | 压缩后 AI 忘了项目结构 | 检索工具随时重新获取 |
| 多人/多 AI 维护不同步 | 各自脑中模型不同 | 共享同一份知识图谱 |
| 读源码文件找函数签名 | read_file 全文件 ~3000 token | outline.py ~100 token |

## 工作流

```
┌─────────────────────────────────────────────────────────┐
│  工作流一：初始化（一次性）                                │
│                                                         │
│  并行子代理分析项目 → 生成 PROJECT_MAP.md                │
│  → 部署 search_map.py → 更新 AGENTS.md                  │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  工作流二：检索（每次改代码前）                            │
│                                                         │
│  search_map.py list → 查章节号 → 搜索关键词              │
│  → 读实际代码确认 → 动手修改                              │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  工作流三：同步（每次改代码后）                            │
│                                                         │
│  代码变更 → 按变更类型表更新对应章节 → 更新日期            │
└─────────────────────────────────────────────────────────┘
```

## 快速开始

### 1. 安装到 Trae IDE

将本目录复制到项目的 `.trae/skills/` 下：

```bash
cp -r ximoAgent-skill .trae/skills/ximoAgent
```

### 2. 让 AI 初始化知识图谱

在 Trae IDE 中对 AI 说：

> "帮我分析项目结构，生成项目知识图谱"

AI 会自动调用此技能，并行分析项目各层面，生成 `PROJECT_MAP.md`，部署检索工具，并更新 `AGENTS.md`。

### 3. 使用检索工具

```bash
# 列出所有章节
python scripts/search_map.py list

# 查看第6章「数据模型」
python scripts/search_map.py 6

# 搜索关键词
python scripts/search_map.py "DI容器"

# 指定其他图谱文件
python scripts/search_map.py list --map docs/ARCHITECTURE.md
```

## 项目结构

```
ximoAgent-skill/
├── SKILL.md                          # 技能说明（AI 读取入口）
├── README.md                         # 本文件
├── LICENSE                           # MIT
├── tools/
│   ├── gen_index.py                  # 项目索引生成器（零依赖，自动分析源码）
│   ├── outline.py                    # 文件大纲提取器（多语言，省 85-95% token）
│   └── search_map.py                 # 知识图谱检索工具（关键词/章节号/鲜度）
└── templates/
    ├── PROJECT_MAP_TEMPLATE.md       # 知识图谱空白模板（10章结构）
    ├── AGENTS_PATCH.md               # AGENTS.md 补丁（渐进式读取规则+工具引导）
    └── INDEX_TEMPLATE.json           # index.json 参考模板
```

## 知识图谱章节结构

`PROJECT_MAP.md` 包含 10 个固定章节 + 1 个自定义章节：

| 章节 | 内容 | 谁填写 |
|------|------|--------|
| 0. 快速导航 | 按意图索引到对应章节 | 模板固定 |
| 1. 架构层次 | 分层架构、启动顺序、设计模式 | AI 分析填写 |
| 2. 目录结构 | 完整目录树 + 文件用途标注 | AI 分析填写 |
| 3. 模块依赖关系 | 依赖链、修改规则、生命周期 | AI 分析填写 |
| 4. 核心业务流程 | 主要业务流程图 | AI 分析填写 |
| 5. 关键函数调用链 | 重要功能的函数调用序列 | AI 分析填写 |
| 6. 数据模型 | ORM 模型、外键约束、数据关系 | AI 分析填写 |
| 7. API 接口定义 | 按功能域分组的接口清单 | AI 分析填写 |
| 8. 项目特有章节 | Agent工具/插件/事件系统等 | AI 按项目定制 |
| 9. 技术栈组件 | 全部依赖及版本约束 | AI 分析填写 |
| 10. 已知风险与禁忌 | 修改禁忌、稳定性风险、技术债 | AI 分析填写 |

## 检索工具特性

### gen_index.py — 项目索引生成器

- **零依赖**：仅需 Python 3.8+
- **多语言支持**：Python, Go, TypeScript/JavaScript, Rust, Java
- **自动语言检测**
- **导出符号提取**：函数、类、类型等
- **Git 热点分析**：识别最近频繁修改的文件
- **鲜度检查**：`--check` 对比源码变更时间
- **JSON 输出**：机器可读，可直接管道给 jq 或其他工具

### outline.py — 文件大纲提取器

- **零依赖**：仅需 Python 3.8+
- **6 语言支持**：Python, Go, TS/JS, Rust, Java
- **四种输出模式**：树形、平铺（行号）、JSON、摘要统计
- **类型过滤**：`--kind=func|class|type|...`
- **递归扫描**：`--recursive` 扫描整个目录
- **省 85-95% token**：vs 读完整文件

### search_map.py — 知识图谱检索

- **零依赖**：仅需 Python 3.8+
- **三种查询模式**：章节列表、章节号精准查、关键词模糊搜
- **输出控制**：`--max-per-section` 限制匹配数、`--no-context` 无上下文
- **忽略空格匹配**：搜索 "DI容器" 能匹配文本中的 "DI 容器"
- **自定义文件路径**：`--map` 参数支持任意 Markdown 文件
- **跨平台**：Windows / macOS / Linux 均可运行

## Token 节省效果

### 三级检索体系（基于本仓库实测）

以下数据基于 project-map-skill 自身代码实测（3 个 Python 文件，共 ~60KB / 1652 行源码），非理论估算：

| 级别 | 工具 | 实测输出量 | 传统方式量 | 节省比例 |
|------|------|-----------|------------|--------|
| 第一级 | `outline.py <file>` | 776 字节 | read_file 全文件 22076 字节 | **省 96.5%** |
| 第一级 | `outline.py src/ -r -s` | 2573 字节（3文件） | read_file 三文件 61127 字节 | **省 95.8%** |
| 第一级 | `search_map.py list` | 671 字节 | read_file 全文 ~12000 字节 | **省 94%+** |
| 第二级 | `search_map.py <章节>` | ~200 字节 | grep + read_file ~3000 字节 | **省 93%** |
| 第二级 | `search_map.py "关键词"` | ~400 字节 | grep + read_file ~4000 字节 | **省 90%** |
| 第三级 | `read_file offset/limit` | ~500 字节 | read_file 全文件 ~3000 字节 | **省 83%** |

> *以上为 project-map-skill 仓库实测数据。实际节省比例随代码风格、注释密度、函数粒度而浮动，但「outline 替代全文件读取」在绝大多数场景下节省 90%+ 是可复现的。*

### 已知限制

- **正则解析精度**：`outline.py` 和 `gen_index.py` 基于正则表达式解析，对 Go 泛型（`func First[T any]`）、TypeScript 复杂类型体操（条件类型、映射类型 `as` 子句）、带复杂返回类型的类方法存在漏报。这是零依赖纯正则方案的有意取舍——覆盖 95% 常见代码，极端泛型语法优先保证不崩溃而非不漏报。在已具备 LSP（语言服务器）的环境中，`outline.py` 的准确度不及 LSP，但其零依赖、即时运行的优势仍在。
- **地图质量天花板**：所有检索效果依赖 `PROJECT_MAP.md` 的质量。AI 生成的地图质量取决于 AI 对项目的理解深度，大型项目建议人工审校关键章节（第4章业务流程、第8章项目特有体系、第10章风险禁忌）。

## 适用场景

### 🟢 高效场景（推荐使用）

| 项目规模 | 文件数 | 推荐度 | 理由 |
|---------|--------|--------|------|
| **大型项目** | 200+ | ⭐⭐⭐⭐⭐ | 全量读取代价极高，地图一次生成长期受益，outline.py 持续节省 90%+ token |
| **中型项目** | 50-200 | ⭐⭐⭐⭐ | outline.py 持续节省 token，检索定位精准，维护成本可控 |
| **接手陌生项目** | 不限 | ⭐⭐⭐⭐⭐ | 快速建立全局认知，避免盲改引入 bug，一次性价值峰值 |
| **多人/多 AI 协作** | 50+ | ⭐⭐⭐⭐ | 共享同一份知识图谱，避免各自脑中模型不同步 |

### 🟡 低效场景（不推荐）

| 项目规模 | 文件数 | 推荐度 | 理由 |
|---------|--------|--------|------|
| **小型项目** | 10-50 | ⭐⭐ | 全量 read_file 代价可接受，地图维护成本占比过高 |
| **微型项目** | <10 | ⭐ | 直接读源码比查地图更快，生成地图是净亏损 |

### 🔴 无效场景（完全无用）

| 场景 | 原因 |
|------|------|
| **一次性脚本/原型** | 生命周期太短，不值得花时间生成和维护地图 |
| **快速迭代期** | 代码每天大改，地图腐化速度远快于同步速度，stale 警告频繁触发反而干扰判断。建议进入稳定迭代期后再启用 |

### 💡 从零开始写项目：分两种用法

从零开始写项目**不是完全无效**，关键看怎么用：

| 用法 | 有效吗 | 说明 |
|------|-------|------|
| **跑 `gen_index.py` / `outline.py` 自动分析** | ❌ 无效 | 空项目没有代码可分析，工具无的放矢，产出的 index.json 只有空结构 |
| **把 `PROJECT_MAP.md` 当设计蓝图手动填写** | ✅ 有效 | 10 章模板本身就是结构化设计清单，动手前先填等于「架构先行」 |

**作为设计蓝图的用法**：新建项目时，先 `cp templates/PROJECT_MAP_TEMPLATE.md PROJECT_MAP.md`，在写第一行代码前逐章填写——

| 章节 | 从零开始时的价值 |
|------|------------------|
| 第 2 章 目录结构 | **先画骨架再填肉**：先定义 `src/domain/`、`src/infra/` 等目录约定，再让 AI 按约定生成代码，比让 AI 随意放文件强得多 |
| 第 3 章 模块依赖 | 明确模块边界和职责，防止 AI 把所有逻辑塞进一个 `utils.py` |
| 第 7 章 API 接口 | 提前规定接口契约，前后端并行开发有据可依 |
| 第 8 章 项目特有体系 | 提前规定错误码规范、日志格式、命名约定——比写完 50 个文件后再统一改成本低得多 |
| 第 9 章 技术栈 | 在动手前锁定依赖选型，避免 AI 半路换框架 |
| 第 10 章 风险禁忌 | 「本项目不用 ORM」「所有 API 必须带 version 前缀」——写在禁忌里，AI 从第一天就不会犯 |

> **本质**：此时 PROJECT_MAP.md 不是「事后分析地图」，而是「事前约束地图」——由人/AI 坐下来先填，而非工具扫描生成。多 AI 并行搭建时，它就是各方的「开发合同」。等项目积累到 20+ 文件后，再叠加 `gen_index.py` 的自动索引能力。

## 兼容性

- **Trae IDE**：作为 Skill 使用，AI 自动调用
- **其他 AI 编程工具**（Cursor、Copilot 等）：手动使用检索工具，将 `SKILL.md` 内容粘贴到系统提示词中
- **纯人工使用**：检索工具本身是独立 Python 脚本，任何人都能用
